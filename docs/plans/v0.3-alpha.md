# v0.3-alpha Plan: Agent-loop Repair and Quality Hardening

## Status

v0.2-alpha shipped the agent-callable foundation:

- agent-facing schemas
- multi-catalog routing
- MCP tools
- framework-neutral tool adapter
- deterministic agent-loop demos
- headphones adapter
- local dogfooding utilities

v0.3-alpha should focus on making RecHarness easier to embed into a reliable recommendation loop.

## Release Theme

Agent-loop repair and quality hardening.

## Product Direction

RecHarness should remain an agent-agnostic recommendation quality layer.

It should not become:

- a full shopping agent
- a checkout system
- a real-time ecommerce crawler
- a hosted production service
- a benchmark report generator for research papers

## Core Goal

Turn RecHarness from a tool that can assist and verify into a deterministic quality loop:

```text
list_catalogs
-> choose explicit domain
-> assist
-> draft answer from bundle
-> verify
-> repair or qualify
-> safe final answer
```

## Must-have Scope

### 1. Deterministic Answer Repair Utility

Add a formal repair layer that can consume verification results and produce a safer answer.

Potential module:

```text
src/recharness/core/answer_repair.py
```

Potential API:

```python
repair_answer_from_verification(
    answer: str,
    verify_response: VerifyResponse,
    assist_response: AssistResponse | None = None,
) -> RepairResult
```

Potential `RepairResult` fields:

```text
status: unchanged | repaired | qualified | failed
repaired_answer: str
changes: list[str]
warnings: list[str]
```

Initial repair behavior should be deterministic and conservative:

- replace hard-constraint-invalid recommendations only when a safe recommended candidate exists
- correct catalog-grounded factual mistakes such as price, stock, ANC, battery, weight, or laptop fit
- add caveats when safe repair is not possible
- preserve cautious wording such as "local catalog indicates" / "本地目录标注"
- never invent product facts

### 2. Agent Loop Contract v2

Add documentation that defines the recommended agent loop.

Potential doc:

```text
docs/agent_loop_contract.md
```

It should explain:

```text
list_catalogs
-> explicit domain selection
-> assist
-> draft
-> verify
-> repair_or_qualify
-> final answer
```

It should also define agent rules:

- do not recommend rejected candidates
- do not claim attributes missing from the catalog
- do not treat soft preferences as hard guarantees
- do not ignore out-of-stock hard exclusions
- do not treat every assist warning as final-answer failure
- always run verification before final answer when using RecHarness in agent mode

### 3. Existing Domain Quality Passes

Improve existing domains only where dogfooding or tests reveal concrete quality gaps.

Focus domains:

```text
backpacks
headphones
```

Do not add a new product domain before the existing two domains are more stable.

Potential headphones quality areas:

- comfort / long-wear / 佩戴舒适
- transparency mode / ambient mode / 通透模式
- multipoint / 多设备连接
- codec / LDAC / AAC / aptX
- over-ear vs in-ear / 头戴 vs 入耳
- avoid confusing microphone noise reduction with ANC

Potential backpacks quality areas:

- business / casual / outdoor / travel scenarios
- capacity range interpretation
- laptop size matching
- water-resistant vs waterproof distinction
- weight and carry comfort claims

### 4. Expanded Edge-case Fixtures

Add edge-case fixtures only to improve quality coverage.

Potential files:

```text
examples/backpacks/dogfooding_tasks.jsonl
examples/headphones/edge_cases.jsonl
```

Coverage should include:

- no product satisfies all hard constraints
- only soft preferences are partially satisfied
- product-name substring collisions
- Chinese / English mixed queries
- budget conflicts
- inventory conflicts
- scenario conflicts
- unsupported claims in agent answers

### 5. Tests for Repair and Edge Cases

v0.3 should include tests for:

- safe unchanged answer
- repaired incorrect claim
- hard-constraint-invalid recommendation
- unable-to-repair answer gets qualified instead of hallucinated
- repair output remains grounded in catalog facts
- demos use the shared repair utility instead of ad-hoc local repair logic

## Should-have Scope

- CLI command for repair, or `verify --repair`
- `docs/repair.md`
- better diagnostics summary for repair results
- demo refactor to use the shared repair utility
- README update describing `assist -> verify -> repair`

## Could-have Scope

Only after must-have scope is stable:

- optional LangGraph integration example
- HTTP service design doc
- optional HTTP prototype

Do not implement HTTP server unless the repair layer and docs are already stable.

## Explicitly Out of Scope

For v0.3-alpha, do not add:

- ecommerce crawling
- checkout
- hosted production service
- external LLM dependency
- vector retrieval
- LLM parser
- paper benchmark reports
- leaderboards
- charts / figures
- new product domain
- generated dogfood or eval outputs committed to the repo

## Proposed Development Sequence

```text
PR 1: Add v0.3-alpha planning doc
PR 2: Define RepairResult schema and deterministic answer repair utility
PR 3: Add repair behavior tests
PR 4: Refactor deterministic demos to use repair utility
PR 5: Add agent loop contract and repair docs
PR 6: Add edge-case fixtures for backpacks/headphones
PR 7: Improve domain adapters only for concrete dogfood failures
PR 8: Update CHANGELOG, ROADMAP, README for v0.3-alpha
```

## Acceptance Criteria for This Planning Task

This planning task is complete when:

1. `docs/plans/v0.3-alpha.md` exists.
2. The document clearly defines v0.3-alpha as "Agent-loop repair and quality hardening."
3. The document prioritizes deterministic repair, agent-loop contract, existing-domain quality, and edge-case fixtures.
4. The document explicitly keeps new domains, HTTP server, vector retrieval, external LLMs, and paper artifacts out of scope.
5. No code, tests, schemas, examples, dependencies, or generated outputs are changed.
